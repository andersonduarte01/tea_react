── FORMATO PADRÃO DE ERRO ──────────────────────────────────────────────────────

  Todos os erros da API seguem o contrato abaixo (handler: apps.api.exceptions):

  400 Validação:
    { "status": "error", "code": "validation_error", "errors": { "campo": ["msg"] } }

  401 Autenticação:
    { "status": "error", "code": "authentication_error", "detail": "..." }

  403 Permissão:
    { "status": "error", "code": "permission_denied",   "detail": "..." }

  404 Não encontrado:
    { "status": "error", "code": "not_found",           "detail": "..." }

  429 Throttle:
    { "status": "error", "code": "throttled", "detail": "...", "retry_after": 60 }

  500 Interno:
    { "detail": "Erro interno do servidor." }

── PAGINAÇÃO ────────────────────────────────────────────────────────────────────

  A maioria dos endpoints de listagem usa paginação por número de página.
  Parâmetros: ?page=<N>  ?page_size=<N>  (padrão: 20, máximo: 100)

  Formato de resposta paginada:
  {
    "count":    248,
    "next":     "http://host/api/v1/paciente/?page=2" | null,
    "previous": null,
    "results":  [ ... ]
  }

  Endpoints SEM paginação (retornam array simples [ ... ]):
    GET profissional/          — dataset limitado ao nº de profissionais da clínica
    GET agenda/profissionais/  — idem; necessário para cross-reference com profissional/
    GET funcionario/           — dataset limitado ao nº de funcionários da clínica

================================================================================
 1. AUTENTICAÇÃO  /api/v1/auth/
================================================================================

Login usa e-mail (não CPF). ROTATE_REFRESH_TOKENS=True: cada chamada a
auth/refresh/ retorna um novo par access+refresh e invalida o anterior.

────────────────────────────────────────────────────────────────────────────────
POST  auth/login/
  Autenticação por e-mail + senha. Retorna tokens e lista de clínicas.

  Request:
    { "email": "usuario@email.com", "password": "senha123" }

  Response 200:
    {
      "access":  "eyJ...",
      "refresh": "eyJ...",
      "usuario": {
        "id":    1,
        "nome":  "Anderson Silva",
        "email": "anderson@email.com",
        "cpf":   "123.456.789-00"  // obtido do perfil vinculado; null se não disponível
      },
      "clinicas": [
        {
          "id":           1,
          "nome":         "Clínica TEA Centro",
          "tipo_usuario": "ADMIN",
          "tela_inicial": "AdminHome"
        }
      ]
    }

  Erros: 401 (credenciais inválidas), 400 (campos ausentes)
  Nota: o app mobile deve usar a lista "clinicas" para exibir o seletor de
  clínica antes de prosseguir. Enviar X-Clinica-ID nos demais endpoints.

────────────────────────────────────────────────────────────────────────────────
POST  auth/refresh/
  Renova o access token usando um refresh token válido.
  ROTATE_REFRESH_TOKENS=True: devolve também um novo refresh token.

  Request:
    { "refresh": "eyJ..." }

  Response 200:
    { "access": "eyJ...", "refresh": "eyJ..." }

  Erros: 401 { "detail": "Token is invalid or expired", "code": "token_not_valid" }

────────────────────────────────────────────────────────────────────────────────
POST  auth/logout/
  Blacklist do refresh token. O access token continua válido até expirar (60 min).

  Request:
    Authorization: Bearer <access_token>
    { "refresh": "eyJ..." }

  Response 204 (sem body)

  Erros: 400 (refresh ausente), 401 (token inválido/já blacklistado)

────────────────────────────────────────────────────────────────────────────────
GET   auth/me/
  Dados do usuário autenticado e suas clínicas ativas.

  Response 200:
    {
      "id":    1,
      "nome":  "Anderson Silva",
      "email": "anderson@email.com",
      "cpf":   "123.456.789-00" | null,
      "clinicas": [ { "id", "nome", "tipo_usuario", "tela_inicial" } ]
    }

────────────────────────────────────────────────────────────────────────────────
POST  auth/change-password/
  Altera a senha do usuário autenticado. Disponível para todos os papéis
  (ADMIN, FUNC, PROF, RESP, PAC).

  Request:
    {
      "senha_atual":    "SenhaAtual@123",   // obrigatório — senha atual do usuário
      "nova_senha":     "NovaSenha@456",    // obrigatório — mínimo 8 caracteres
      "confirmar_nova": "NovaSenha@456"     // obrigatório — deve ser igual a nova_senha
    }

  Response 204 (sem body) — senha alterada com sucesso.

  Erros:
    400 {
          "status": "error",
          "code":   "validation_error",
          "errors": {
            "senha_atual":    ["Senha atual incorreta."],
            "nova_senha":     ["Certifique-se de que este campo tenha no mínimo 8 caracteres.",
                               "A nova senha não pode ser igual à senha atual."],
            "confirmar_nova": ["As senhas não coincidem."]
          }
        }
    401 { "status": "error", "code": "authentication_error", "detail": "..." }

      "meu_papel":         "ADMIN",

      "tipo_usuario_atual": "ADMIN"   // alias de meu_papel para o app mobile
    }

────────────────────────────────────────────────────────────────────────────────
PATCH clinica/
  Atualização parcial dos dados cadastrais da clínica. Restrito a ADMIN.
  Retorna 403 para qualquer outro papel (FUNC, PROF, RESP, PAC).

  Request body (todos os campos opcionais — PATCH parcial):
    {
      "nome":     "Clínica TEA Centro",   // não pode ser vazio se enviado
      "telefone": "85333330000",
      "email":    "contato@clinica.com",  // deve ser e-mail válido se enviado
      "endereco": {
        "logradouro":  "Rua das Flores",  // partial update — enviar apenas o que mudar
        "numero":      "100",
        "complemento": "Sala 2",
        "bairro":      "Centro",
        "cidade":      "Fortaleza",
        "estado":      "CE",              // exatamente 2 caracteres se enviado
        "cep":         "60000-000"        // 8 dígitos após remover máscara; normalizado antes de persistir
      }
    }

  Regras:
    • Se "endereco" for omitido, o endereço existente não é alterado.
    • Se "endereco" for enviado, partial update nos subcampos (apenas os
      enviados são atualizados; os demais permanecem inalterados).
    • "cep" normalizado: máscara removida, apenas dígitos armazenados
      ("60000-000" → "60000000").
    • Campos somente leitura (ignorados mesmo se enviados):
      id, slug, cnpj, foto, ativa, data_criacao.

  Response 200 — objeto completo da clínica (mesmo shape do GET clinica/):
    { "id", "nome", "slug", "cnpj", "telefone", "email", "foto", "endereco",
      "ativa", "data_criacao", "meu_papel", "meu_papel_display", "tipo_usuario_atual" }

  Erros:
    400 { "status": "error", "code": "validation_error", "errors": { "campo": ["msg"] } }
    403 { "status": "error", "code": "permission_denied",
          "detail": "Apenas administradores podem editar a clínica." }

────────────────────────────────────────────────────────────────────────────────
PATCH clinica/foto/
  Substitui a foto da clínica. Restrito a ADMIN.
  Content-Type: multipart/form-data

  Request (form-data):
    foto  — arquivo de imagem (obrigatório, max 5 MB)
            Processado automaticamente: redimensionado para 300×300 px, salvo
            como JPEG com qualidade 85 (imagekit ProcessedImageField).

  Response 200:
    { "foto": "https://host/media/clinica/fotos/abc123.jpg" }

  Erros:
    400 { "status": "error", "code": "validation_error",
          "errors": { "foto": ["Nenhum arquivo foi submetido."] } }
    400 { "status": "error", "code": "validation_error",
          "errors": { "foto": ["A foto não pode ultrapassar 5 MB."] } }
    403 { "status": "error", "code": "permission_denied",
          "detail": "Apenas administradores podem alterar a foto da clínica." }

  Nota mobile: após o PATCH bem-sucedido, a nova URL retornada deve substituir
  o campo "foto" no estado local da tela. O GET clinica/ também retorna
  a URL atualizada (campo "foto").

 4. PROFISSIONAL  /api/v1/profissional/
================================================================================

────────────────────────────────────────────────────────────────────────────────
GET   profissional/
  Lista profissionais ativos da clínica.
  Permitido: ADMIN, PROF, FUNC.
  Filtros: ?search=<nome>  |  ?ordering=nome

 ── Agendamentos ──────────────────────────────────────────────────────────────

GET    agenda/agendamentos/
  Lista agendamentos com paginação.

  Visibilidade por papel:
    ADMIN/FUNC : todos os agendamentos da clínica
    PROF       : apenas os seus próprios agendamentos
    RESP       : agendamentos dos seus pacientes vinculados
    PAC        : apenas os seus próprios

  ── FILTRO DE STATUS (obrigatório entender antes de usar) ───────────────────

  Por padrão, a listagem EXCLUI agendamentos com status "cancelado" e
  "remarcado". Isso evita poluição na tela principal do app mobile.
  Para acessá-los use ?status= conforme tabela abaixo.

  ?status=<valor>   Comportamento
  ─────────────────────────────────────────────────────────────────────────────
  (sem parâmetro)   Retorna: aguardando + realizado + nao_compareceu
                    Exclui : cancelado, remarcado

  aguardando        Todos os agendamentos aguardando atendimento,
                    incluindo os atrasados e os confirmados.
                    Use quando quiser a fila completa do dia.

  atrasado          Agendamentos com status=aguardando cuja data/hora já
                    passou. Calculado em tempo real pelo servidor — não
                    depende do campo status_operacional no banco.
                    Use na tela de pendências/alertas do app.

  cancelado         Apenas agendamentos cancelados.
                    Use no histórico de cancelamentos ou auditoria.

  remarcado         Apenas agendamentos que foram remarcados.
                    O campo "origem_remarcacao" indica o agendamento
                    anterior na cadeia de remarcação.

  realizado         Apenas os realizados.

  nao_compareceu    Apenas os com não comparecimento registrado.

  todos             Retorna absolutamente tudo, sem nenhuma exclusão.
                    Use com cuidado — pode gerar payloads grandes.

  Exemplo — tela de pendências (atrasados):
    GET agenda/agendamentos/?status=atrasado

  Exemplo — histórico de cancelamentos de um paciente:
    GET agenda/agendamentos/?status=cancelado&paciente_id=42

  Exemplo — agenda do dia de um profissional:
    GET agenda/agendamentos/?status=aguardando&profissional_id=7&data_inicio=2026-05-24&data_fim=2026-05-24

  ── DEMAIS FILTROS ──────────────────────────────────────────────────────────

  ?data_inicio=YYYY-MM-DD   Filtra a partir desta data (inclusive)
  ?data_fim=YYYY-MM-DD      Filtra até esta data (inclusive)
  ?profissional_id=<id>     Filtra por profissional
  ?paciente_id=<id>         Filtra por paciente
  ?page=<N>                 Paginação (padrão: 20 por página)

  ── SHAPE DA RESPOSTA (item da lista) ───────────────────────────────────────

  {
    "id":                42,
    "paciente_id":       10,
    "paciente_nome":     "João Silva",
    "profissional_nome": "Dr. Carlos Lima",
    "data":              "2026-05-24",
    "hora_inicio":       "14:00:00",
    "hora_fim":          "14:50:00",
    "status":            "aguardando",          // status clínico persistido
    "status_operacional": "atrasado",           // calculado em tempo real — ver nota
    "visual_status": {
      "label":  "Aguardando",
      "classe": "slot-agendado slot-atrasado",  // slot-atrasado = alerta visual
      "icone":  "fa-clock-o",
      "alerta": true                            // calculado em tempo real — ver nota
    }
  }

  Nota sobre status_operacional e visual_status.alerta:
    Ambos os campos são calculados em tempo real pelo serializer no momento
    da resposta — não dependem do valor persistido no banco. A regra é:
    se status == "aguardando" e data+hora_inicio < agora → status_operacional
    retorna "atrasado" e visual_status.alerta retorna true, independente do
    que estiver gravado no campo status_operacional do banco.

    Isso garante que o app receba o estado correto mesmo quando o job de
    atualização periódica do banco estiver defasado.

    Valores possíveis de status_operacional:
      "normal"                — agendamento dentro do prazo
      "confirmado"            — presença confirmada via POST confirmar/
      "atrasado"              — hora já passou e ainda aguardando
      "pendente_reagendamento"— marcado para reposição

    Campos opcionais em visual_status (presentes apenas quando aplicável):
      "alerta": true          — agendamento atrasado
      "confirmado": true      — presença confirmada, ainda aguardando
      "pendente_reagendamento": true

  ⚠ ATENÇÃO FRONTEND — cross-reference entre endpoints:
    "id"             = ID da ConfiguracaoAgenda  (NÃO é o ID do Profissional)
    "profissional_id" = ID do Profissional        (use este para cruzar com profissional/<id>/)
    Para saber quais profissionais NÃO têm agenda: buscar profissional/ e excluir os
    cujo "id" esteja presente em "profissional_id" desta listagem.
    NUNCA cruzar pelo campo "id" — ele identifica a configuração, não o profissional.

POST   agenda/profissionais/
  Cria configuração de agenda para um profissional. Restrito a ADMIN.

  Response da listagem — shape de CADA ITEM:
    {
      "id":                        10,    // ← ID da ConfiguracaoAgenda (não do profissional)
      "profissional_id":           42,    // ← ID do Profissional — use este para cross-reference
      "nome":                      "Dr. Carlos Lima",
      "funcao":                    "Psiquiatra",
      "foto":                      "https://..." | null,
      "hora_inicio":               "08:00:00",
      "hora_fim":                  "18:00:00",
      "duracao_sessao":            50,
      "intervalo":                 10,
      "atende_feriados_nacionais": false, // bool, nunca null
      "atende_feriados_locais":    false, // bool, nunca null
      "dias_trabalho":             { "segunda": true, "terca": true, "quarta": true,
                                     "quinta": true, "sexta": true, "sabado": false, "domingo": false }
    }

  Request body (POST):
    {
      "profissional_id":           42,       // obrigatório — deve pertencer à clínica
      "hora_inicio":               "08:00",  // obrigatório — HH:MM
      "hora_fim":                  "18:00",  // obrigatório — HH:MM; deve ser > hora_inicio
      "duracao_sessao":            50,       // obrigatório — minutos (mín. 1)
      "intervalo":                 10,       // obrigatório — minutos entre sessões (mín. 0)
      "dias_trabalho": {                     // obrigatório — todos os 7 campos booleanos
        "segunda": true,  "terca": true,  "quarta": true,
        "quinta":  true,  "sexta": true,  "sabado": false,  "domingo": false
      },
      "hora_almoco_inicio":        "12:00",  // opcional — HH:MM ou null
      "hora_almoco_fim":           "13:00",  // opcional — HH:MM ou null
      "atende_feriados_nacionais": false,    // opcional, default false
      "atende_feriados_locais":    false     // opcional, default false
    }

  Regras:
    • hora_almoco_inicio e hora_almoco_fim devem ser ambos informados ou ambos omitidos.
    • Operação atômica: ConfiguracaoAgenda + DiasTrabalho criados juntos.

  Response 201 — objeto completo (mesmo shape do GET agenda/profissionais/<id>/).

  Comportamento para profissional com agenda inativa (ativa=False):
    Se já existe um ConfiguracaoAgenda com ativa=False para o profissional,
    o POST reativa esse registro e atualiza todos os seus campos com os dados
    do request body (reativar + atualizar). DiasTrabalho também é atualizado.
    Retorna 201 com o objeto reativado — sem criar duplicata no banco.

  Erros:
    400 { "status": "error", "code": "validation_error", "errors": { ... } }
    403 { "status": "error", "code": "permission_denied", "detail": "..." }
    404 not_found (profissional_id não encontrado ou inativo nesta clínica)
    409 { "status": "error", "code": "conflict",
          "detail": "Este profissional já possui uma agenda configurada nesta clínica." }
        Ocorre SOMENTE quando já existe ConfiguracaoAgenda com ativa=True.
        Se existe config com ativa=False → reativa (201). Sem config → cria (201).

  Nota interna — signal post_save:
    O model ConfiguracaoAgenda possui um signal post_save (signals.py) que cria
    DiasTrabalho automaticamente com get_or_create a cada nova ConfiguracaoAgenda.
    O view usa update_or_create para DiasTrabalho (não create) justamente por isso.
    Nunca usar DiasTrabalho.objects.create() neste contexto — causa IntegrityError.

POST   agenda/profissionais/limpar-orfaos/
  Executa soft delete nas ConfiguracaoAgenda órfãs da clínica. Restrito a ADMIN.

  Critério de órfão: ConfiguracaoAgenda cujo profissional_id não possui
  UsuarioClinica com ativo=True para a clínica do X-Clinica-ID.

  Sem body.

  Response 200:
    {
      "registros_inativados": 2,       // quantidade de configs marcadas como ativo=False
      "detalhes": [
        {
          "configuracao_id": 28,
          "profissional_id": 15,
          "profissional_nome": "Dr. Fulano"
        }
      ]
    }

  Response 200 (nenhum órfão encontrado):
    { "registros_inativados": 0, "detalhes": [] }

  Erros:
    403 { "status": "error", "code": "permission_denied", "detail": "..." }

  Nota: operação segura e reversível — os registros recebem ativo=False
  (soft delete) e não são removidos do banco. O GET agenda/profissionais/
  já filtra ativo=True, portanto os configs inativadas somem da listagem
  imediatamente. Para consultar o efeito, execute GET agenda/profissionais/
  antes e depois da chamada.

  Uso recomendado: chamar uma vez após detectar divergência entre a contagem
  de GET agenda/profissionais/ e o número real de profissionais ativos.

 RESUMO DE PERMISSÕES
================================================================================

 ADMIN   — Acesso total. Único que pode usar PATCH clinica/ (edição de dados
           cadastrais), GET funcionario/<id>/, POST profissional/ e
           POST funcionario/ (criação via app mobile).
 FUNC    — Acesso administrativo semelhante ao ADMIN (leitura).
           Pode criar/gerenciar agendamentos e registrar faltas.
 PROF    — Acessa seus agendamentos, sessões e pacientes.
           Em GET profissional/<id>/: apenas o próprio ID.
 RESP    — Apenas seus pacientes. Pode ver agendamentos e sessões
           não-privadas. Pode criar eventos e responder formulários.
 PAC     — Apenas seus próprios dados. Pode responder formulários.

================================================================================
 OBSERVAÇÕES TÉCNICAS
================================================================================

 Token lifetime:
   access : 60 minutos
   refresh: 30 dias (rotacionado a cada uso)

 CPF no login:
   O campo "cpf" em auth/login/ vem do perfil vinculado ao usuário
   (Profissional > Funcionario > Paciente). Retorna null para usuários
   sem perfil com CPF (ex: administrador puro sem perfil de profissional).

 Campo "ativo" em funcionario/<id>/:
   O model Funcionario não possui campo "ativo". O valor é obtido de
   UsuarioClinica.ativo — indica se o vínculo com a clínica está ativo.

 chamados_abertos / relatorios_pendentes:
   Módulos não implementados. Retornam sempre 0 no dashboard.
